# Multica

**Multica** is an open-source managed AI coding agents platform. Assign tasks to coding agents (Claude Code, Codex, GitHub Copilot CLI, and others) the same way you'd assign them to a human teammate — agents pick up work autonomously, write code, report blockers, and compound skills over time.

## Overview

Deploy your own Multica instance with this production-ready Dokploy template featuring:

- **PostgreSQL 17 + pgvector** for structured data and vector embeddings
- **Cloudflare R2 integration** for scalable file storage (optional, falls back to local volume)
- **Two Traefik routes** — frontend UI and backend API each on their own domain
- **WebSocket-optimized backend** — `flushinterval=-1` ensures persistent daemon connections don't stall
- **Signup access control** — allowlist by email or domain for private team instances

## Architecture

```
Internet
   │
   ├──[DOMAIN]──────────────▶ frontend (Next.js, :3000)
   │                               │
   └──[API_DOMAIN]─────────▶ backend (Go, :8080)
                                   │ /ws (WebSocket)
                                   │ /api/...
                                   │
                              multica-net (internal)
                                   │
                             postgres (:5432)
                           pgvector/pgvector:pg17


   [optional] Cloudflare R2 ◀── backend (file uploads)
```

## Services

| Service | Image | Role |
|---------|-------|------|
| `frontend` | `ghcr.io/multica-ai/multica-web:v0.2.27` | Next.js web UI |
| `backend` | `ghcr.io/multica-ai/multica-backend:v0.2.27` | Go API + WebSocket server |
| `postgres` | `pgvector/pgvector:pg17` | Database with vector support |

## Requirements

- **CPU**: 1+ vCPU
- **RAM**: 1 GB minimum, 2 GB recommended
- **Disk**: 10 GB (scales with uploaded files — use R2 to offload)
- **Two DNS records**: one for `DOMAIN` (frontend), one for `API_DOMAIN` (backend)
- **Docker Compose v2.x+** (bundled with Dokploy)

## Domain Setup

Multica requires **two separate domains** (or subdomains):

| Variable | Purpose | Example |
|----------|---------|---------|
| `DOMAIN` | Frontend web UI | `multica.example.com` |
| `API_DOMAIN` | Backend API + WebSocket | `api.multica.example.com` |

> **Why two domains?** `NEXT_PUBLIC_WS_URL` is compiled into the pre-built `multica-web` image at build time as a Docker `ARG` — it cannot be injected at runtime. The frontend defaults `REMOTE_API_URL` to the internal `http://backend:8080` service name (REST API proxying works inside the Docker network), but `NEXT_PUBLIC_WS_URL` has no default and must be a publicly reachable `wss://` URL for the browser's WebSocket connection to work. **Path-based routing (e.g. `example.com` + `example.com/api`) is not supported with the pre-built image** — two separate domains is the only valid setup without a custom build.

Point both DNS records to your Dokploy server IP. Traefik provisions separate SSL certificates for each via Let's Encrypt.

### Cookie Domain (cross-subdomain sessions)

When `DOMAIN` and `API_DOMAIN` share a parent domain (e.g. `multica.example.com` and `api.multica.example.com`), set `COOKIE_DOMAIN` to the shared parent:

```
COOKIE_DOMAIN = .example.com
```

Leave blank for single-hostname or local deployments.

## Configuration Variables

### Required

| Variable | Description |
|----------|-------------|
| `DOMAIN` | Frontend domain (e.g. `multica.example.com`) |
| `API_DOMAIN` | Backend API domain (e.g. `api.multica.example.com`) |
| `POSTGRES_PASSWORD` | Database password (auto-generated by Dokploy) |
| `JWT_SECRET` | JWT signing secret (auto-generated by Dokploy) |

### Email Authentication

Multica uses magic-link / OTP email authentication. Without a Resend key, verification codes print to the backend container logs — fine for private instances or testing.

| Variable | Default | Description |
|----------|---------|-------------|
| `RESEND_API_KEY` | _(empty)_ | [Resend](https://resend.com) API key for sending emails |
| `RESEND_FROM_EMAIL` | `noreply@multica.ai` | Sender address (must be verified in Resend) |

### Google OAuth (optional)

| Variable | Description |
|----------|-------------|
| `GOOGLE_CLIENT_ID` | Google OAuth client ID |
| `GOOGLE_CLIENT_SECRET` | Google OAuth client secret |

The redirect URI is automatically `https://${DOMAIN}/auth/callback` — register this in your Google Cloud console.

### Signup Control

| Variable | Default | Description |
|----------|---------|-------------|
| `ALLOW_SIGNUP` | `true` | Set to `false` to disable new registrations |
| `ALLOWED_EMAIL_DOMAINS` | _(empty)_ | Comma-separated allowed domains (e.g. `company.com`) |
| `ALLOWED_EMAILS` | _(empty)_ | Comma-separated allowlist of exact email addresses |

> **Recommendation**: Set `ALLOW_SIGNUP=false` after creating your team accounts on any internet-facing instance.

### Cloudflare R2 Storage (optional)

Without R2 configured, uploaded files are stored in the `backend-uploads` Docker volume on the host. Configure R2 to store files off-server and survive container replacements.

| Variable | Description |
|----------|-------------|
| `AWS_ENDPOINT_URL` | `https://<ACCOUNT_ID>.r2.cloudflarestorage.com` |
| `S3_BUCKET` | R2 bucket name |
| `S3_REGION` | Always `auto` for R2 |
| `AWS_ACCESS_KEY_ID` | R2 API access key ID |
| `AWS_SECRET_ACCESS_KEY` | R2 API secret access key |

> **Note**: `CLOUDFRONT_DOMAIN` is not set in this template — it is AWS CloudFront-specific and incompatible with R2. Files are served directly from the R2 endpoint when a custom CDN domain is not configured.

### Advanced

| Variable | Default | Description |
|----------|---------|-------------|
| `COOKIE_DOMAIN` | _(empty)_ | Shared cookie domain for cross-subdomain sessions |
| `CORS_ALLOWED_ORIGINS` | _(FRONTEND_ORIGIN)_ | Additional CORS origins (comma-separated) |
| `POSTGRES_DB` | `multica` | Database name |
| `POSTGRES_USER` | `multica` | Database user |

## Cloudflare R2 Setup

### 1. Create an R2 bucket

1. Go to [Cloudflare Dashboard](https://dash.cloudflare.com) → **R2** → **Overview**
2. Click **Create bucket**
3. Name it (e.g. `multica-uploads`) — note the name for `S3_BUCKET`

### 2. Generate R2 API credentials

1. In R2 → **Overview**, click **Manage R2 API Tokens**
2. Click **Create API token**
3. Set permissions: **Object Read & Write**
4. Optionally restrict to your specific bucket
5. Copy the **Access Key ID** → `AWS_ACCESS_KEY_ID`
6. Copy the **Secret Access Key** → `AWS_SECRET_ACCESS_KEY`

### 3. Get your Account ID

Your Account ID appears in the Cloudflare Dashboard URL and on the **Overview** page right-hand panel:

```
AWS_ENDPOINT_URL = https://<ACCOUNT_ID>.r2.cloudflarestorage.com
```

### 4. Configure in Dokploy

Set all four R2 variables in the Dokploy UI. Restart the backend service to apply.

## Deployment

### 1. Deploy via Dokploy

1. In Dokploy, create a new **Compose** project
2. Select this template or paste the `docker-compose.yml`
3. Fill in the required variables (Dokploy auto-generates `POSTGRES_PASSWORD` and `JWT_SECRET`)
4. Set `API_DOMAIN` to your backend subdomain (e.g. `api.multica.example.com`)
5. Deploy

### 2. Verify DNS

Confirm both domains resolve to your server before Traefik attempts certificate provisioning:

```bash
dig +short multica.example.com
dig +short api.multica.example.com
```

Both should return your server's IP.

### 3. Check services are healthy

```bash
docker compose -p multica ps
```

Expected: all three services `healthy` within ~90 seconds of first boot (backend runs DB migrations on startup).

### 4. Create your first account

Open `https://DOMAIN` in a browser. If `RESEND_API_KEY` is not set, check backend logs for the verification code:

```bash
docker logs multica-backend-1 | grep "Verification code"
```

In Dokploy, find the container name under **Services → multica → backend → Logs**, or run the command above from the host terminal.

## Post-Deployment

### Connect the CLI daemon

The Multica CLI daemon runs on developer machines and connects to your server's WebSocket endpoint. After installing the CLI:

```bash
multica setup self-host
# Enter your server URL when prompted: https://API_DOMAIN
```

The daemon uses `wss://API_DOMAIN/ws` for real-time task coordination.

### Lock down signups

After creating your team accounts:

```yaml
ALLOW_SIGNUP: "false"
```

Or restrict to your company domain:

```yaml
ALLOWED_EMAIL_DOMAINS: "yourcompany.com"
```

### Upgrading

Update image tags in `docker-compose.yml` and redeploy. Check the [Multica releases](https://github.com/multica-ai/multica/releases) page for migration notes between versions.

## Troubleshooting

### WebSocket connections dropping

The Multica daemon maintains persistent WebSocket connections on `/ws`. If connections drop frequently:

- Verify `traefik.http.services.multica-api.loadbalancer.responseforwarding.flushinterval: "-1"` is applied (check `docker inspect` on the backend container)
- For Traefik idle timeout: configure `--entrypoints.websecure.transport.respondingTimeouts.idleTimeout=86400` in your Traefik static config
- The daemon heartbeat is every 15 seconds — any proxy idle timeout below ~30s will cause disconnections

### Verification codes not arriving

If Resend is not configured, codes print to backend logs:

```bash
docker logs multica-backend-1 2>&1 | tail -50
```

For production email delivery, set `RESEND_API_KEY` and ensure `RESEND_FROM_EMAIL` is a verified sender in Resend.

### Backend health check failing

The backend runs database migrations on first boot, which can take 30–60 seconds. The health check has `start_period: 60s` to accommodate this. If it still fails:

```bash
docker logs multica-backend-1 2>&1 | tail -50
```

Common causes: wrong `DATABASE_URL` (check `POSTGRES_PASSWORD` matches between services), or postgres not yet healthy.

### Frontend can't reach the backend API

Verify:
1. `API_DOMAIN` DNS resolves correctly
2. Traefik has provisioned an SSL certificate for `API_DOMAIN` (check Traefik logs)
3. `NEXT_PUBLIC_API_URL` and `NEXT_PUBLIC_WS_URL` in the frontend match `API_DOMAIN`

### R2 uploads failing

- Confirm `AWS_ENDPOINT_URL` uses your **Account ID**, not the bucket name
- Ensure the R2 API token has **Object Read & Write** on the target bucket
- `S3_REGION` must be `auto` for R2 (not `us-east-1` or any other region)
